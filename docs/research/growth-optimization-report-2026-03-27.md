# boby-grow-weapp 中国微信场景增长优化调研与竞品分析报告（决策版）

- 版本：v1.0
- 日期：2026-03-27
- 调研目标：围绕“留存与活跃”，给出界面优化、导航改造、功能去冗余与新增功能的可执行方案。
- 适用项目：`E:\project\must\boby-grow-weapp`

---

## 1. 结论先行（可直接用于评审）

### 1.1 核心判断
1. 当前项目具备较完整业务链路（建档-任务-提交-审核-奖励-成长），但**入口分散、功能重复、反馈闭环不足**，导致用户“能用但不黏”。
2. 父母端主导航仅 3 项（工作台/任务中心/孩子管理），与实际高频任务不匹配：审核消息、设置聚合入口缺失，造成路径跳转成本高。
3. 孩子端“愿望提交”存在明显重复能力（`child/home` 已内嵌抽屉 + `child/submit-wish` 独立页），应统一为单一路径。
4. 市场层面（中国微信场景）足够大，且“轻量家庭激励 + 低门槛协作”仍有空位；竞品普遍在“成长反馈解释性、家庭协作效率、审核效率”上存在短板。
5. 推荐优先策略：**先做结构减法（去冗余 + 导航收敛）再做机制加法（周报、连续挑战、提醒、奖励闭环）**。

### 1.2 决策建议（最终）
- 父母端导航推荐采用 5 主项：`总览 / 任务 / 审核消息 / 孩子 / 我的`。
- 将 `setting` 与 `family-manage` 合并为一个“我的”聚合入口，禁止双入口并行。
- 将 `bind-guide`（绑定流程）与 `mcp-verify`（查询码）进行职责拆分与路由重构：
  - 绑定流程固定在“孩子详情-绑定管理”
  - 查询码能力归入“我的-安全与隐私”
- 删除孩子端独立 `submit-wish` 主入口，仅保留 `child/home` 内嵌愿望提交。
- 新增三类“留存型”能力：成长周报、连续达标挑战、家庭协作提醒。

## 2. 项目基线诊断（当前仓库）

## 2.1 全链路地图（父母端/孩子端）

### 父母端主链路
- 引导与建档：`auth/* -> parent/profile-edit (onboarding) -> add-child`。
- 任务配置：`parent/task-manage -> parent/add-task`。
- 审核与反馈：`parent/dashboard` 查看待审计计数，进入 `parent/audit-center` 审核。
- 奖励与成长观察：`parent/child-profiles` 查看完成结构、时间线与成长信号。

### 孩子端主链路
- 日常执行：`child/home` 查看任务并提交（支持反思文本）。
- 奖励申请：`child/rewards` 提交兑换。
- 审核状态：`child/submit-manage` 查看状态。
- 愿望提交：`child/home` 抽屉 + `child/submit-wish` 独立页（重复）。

## 2.2 信息架构与导航问题清单

1. 主导航承载不足：
- 当前 tabBar 仅 3 项（见 `miniprogram/app.json`），但父母端实际高频存在“审核、设置、孩子”等多类入口，造成跨层跳转。

2. 设置信息架构重复：
- `parent/setting` 与 `parent/family-manage` 都承载“协管、孩子档案、每日上限、MCP、个人资料”等设置项，属于结构性重复。

3. 任务闭环入口不统一：
- 审核动作与消息提醒未独立收敛到一个“消息/审核中心”，用户需要从 dashboard 卡片进入，易漏审。

4. 孩子端愿望路径重复：
- `child/home` 已内嵌愿望提交流程，`child/submit-wish` 仍独立存在，属于同能力多入口。

## 2.3 功能冗余清单（重点核查项）

| 冗余点 | 当前表现 | 风险 | 建议 |
|---|---|---|---|
| `setting` vs `family-manage` | 菜单项高度重合 | 用户认知负担、维护成本翻倍 | 合并为“我的”单入口 |
| `bind-guide` vs `mcp-verify` | 都与核验码/绑定相关 | 概念混淆（绑定 vs 查询） | 绑定归孩子域，查询归安全域 |
| `child/home` vs `child/submit-wish` | 愿望提交双入口 | 数据分流、路径冲突 | 只保留 home 内嵌入口 |

## 3. 市场规模（中国微信场景）

> 口径说明：采用 Top-down + Bottom-up 双口径，并给出区间估算而非单点拍脑袋。

## 3.1 Top-down（行业总盘）

- 中国网民规模（截至 2025-06）：**11.23 亿**，互联网普及率 **79.7%**（CNNIC 第56次报告）。
- 微信及 WeChat 合并月活（截至 2025-12）：**14.18 亿**（腾讯 2025 年报）。
- 小学在校生（2024 学年）：**10,584.37 万人**（教育部统计公报）。

推论：
- 家庭教育与亲子激励工具的“父母端基础触达池”足够大，微信作为分发与触达基础设施成立。

## 3.2 Bottom-up（目标家庭可触达盘）

### 假设
1. 目标家庭：有 6-12 岁孩子（对应小学阶段）。
2. 以小学在校生近似“目标孩子数”：10,584.37 万。
3. 估算目标家庭数区间：
- 保守：按 1.2 个目标年龄孩子/家庭 -> 8,820 万家庭
- 中性：按 1.1 个目标年龄孩子/家庭 -> 9,622 万家庭
- 乐观：按 1.0 个目标年龄孩子/家庭 -> 10,584 万家庭

4. 微信可触达率（家长端）按 85%-95% 区间。

### 估算结果
- SAM（可触达家庭）约：**7,497 万 - 10,055 万家庭**。
- SOM（3年可达，按 0.3%-1.0% 渗透）约：**22.5 万 - 100.6 万家庭**。

结论：
- 该项目若打磨好“日活循环 + 审核效率 + 成长反馈”，具备成长为中等规模细分产品的空间。

## 6. 导航改造与路由治理（两套方案）

## 6.1 方案A（推荐）

### 父母端 tabBar（5项）
- 总览（dashboard）
- 任务（task-manage）
- 审核消息（audit-center + 待办聚合）
- 孩子（child-manage）
- 我的（setting 聚合）

### 优势
- 与高频心智一致，减少跨层跳转。
- 审核能力上升为主导航，直接拉动“审核处理时长”指标。
- 设置能力单点收敛，利于长期扩展。

### 风险
- 主导航从 3->5，首次认知成本略升；需配首周引导。

## 6.2 方案B（备选）

### 父母端 tabBar（4项）
- 总览 / 任务 / 孩子 / 我的
- 审核消息放在总览顶部常驻消息中心

### 优势
- 导航更精简。

### 风险
- 审核入口仍可能被弱化，漏审风险高于方案A。

## 6.3 最终建议
- 采用方案A，首发即 5 项主导航。

## 8. 公共接口 / 导航 / 埋点建议

## 8.1 导航与路由
- `app.json` 从 3 tab 扩展为 5 tab（父母端）。
- 统一路由原则：同一能力仅保留 1 个主入口，其他位置仅保留“快捷跳转”。

## 8.2 API 建议新增
1. `task.getPendingAuditSummary`
- 用途：返回待审核聚合计数（任务/奖励/愿望），驱动导航红点与收件箱排序。

2. `point.getRedeemHistory`
- 用途：孩子端查看兑换历史，补齐奖励闭环。

3. `growth.getStreakAndWeeklyDigest`
- 用途：返回连续达标天数 + 周报摘要，支持留存功能。

## 8.3 事件埋点建议新增
- `onboarding_step_completed`
- `task_cycle_closed`
- `audit_turnaround_time`
- `reward_loop_closed`

统一字段：`role/childId/familyId/ts/source`。

## 9. 可执行实验方案（含验收阈值）

## 9.1 核心指标定义
1. D7 家庭留存率
- 定义：首日激活家庭在第7天仍有任一关键行为（任务提交/审核/兑换）。

2. 周活跃家庭（WAF）
- 定义：最近7天至少发生 2 次关键行为的家庭数。

3. 任务闭环率
- 定义：`已提交且被审核完成任务 / 应完成任务`。

4. 审核处理时长
- 定义：从孩子提交到家长审核完成的中位时长（小时）。

## 9.2 实验设计
- 方案：分阶段灰度（10% -> 30% -> 100%），对照旧导航与新导航。
- 周期：每阶段至少 7 天，完整观察 4-6 周。
- 分层：新家庭 vs 老家庭；1娃家庭 vs 多娃家庭。

## 9.3 默认验收阈值（相对提升）
- D7 家庭留存：+15%（相对）
- 周活跃家庭：+12%（相对）
- 任务闭环率：+18%（相对）
- 审核处理时长：-25%（相对下降）

> 注：最终阈值需结合上线前 2 周基线复核后锁定。

## 10. 合规与风险提示

1. 儿童信息处理必须以监护人授权为前提；14岁以下儿童信息需专门保护。
2. 绑定与查询码能力应分域治理，避免误解为“可操作积分”。
3. 埋点与日志避免采集不必要敏感字段，遵循最小化原则。
